  ## A feature can work and still be badly wired

  ```ts
  type StorageProvider = "aws" | "sftp" | "webdav";
  
  interface Attachment {
    id: string;
    localPath: string;
    destination: StorageProvider;
  }
  
  class AttachmentService {
    async upload(
      attachment: Attachment,
      awsKey?: string,
      sftpHost?: string,
      webDavUrl?: string,
    ) {
      if (attachment.destination === "sftp") {
        return this.uploadToSftp(attachment, sftpHost!);
      }
  
      if (attachment.destination === "webdav") {
        return this.uploadToWebDav(attachment, webDavUrl!);
      }
  
      return this.uploadToAws(attachment, awsKey!);
    }
  }
  ```

  Dependency injection usually starts with discomfort, not with a framework.
  The code above can upload a file, but it is solving too many problems in one
  place.

  - One class owns the upload logic and the storage-selection logic.
  - The caller has to know which optional config matters for which branch.
  - The decision of which dependency to use happens deep inside the workflow.

  That is the core smell DI is trying to remove. The goal is not "use a
  container". The goal is "stop hard-wiring decisions into the object that just
  wants to do its job."

  ## First extract the behavior that changes

  ```ts
  interface AttachmentStorage {
    upload(attachment: Attachment): Promise<string>;
  }
  
  class AwsStorage implements AttachmentStorage {
    async upload(attachment: Attachment): Promise<string> {
      return attachment.id;
    }
  }
  
  class SftpStorage implements AttachmentStorage {
    async upload(attachment: Attachment): Promise<string> {
      return attachment.id;
    }
  }
  
  class WebDavStorage implements AttachmentStorage {
    async upload(attachment: Attachment): Promise<string> {
      return attachment.id;
    }
  }
  ```

  DI gets much less mystical once the changing part is named.

  `AttachmentService` does not really care about AWS, SFTP, or WebDAV. It cares
  that *something* knows how to store an attachment. That is why the abstraction
  comes first.

  - Business code depends on a capability, not a vendor.
  - Each implementation keeps only the config it actually needs.
  - The runtime decision is now free to move somewhere else.

  ## Injection just means passing the dependency in

  ```ts
  interface Company {
    storage: StorageProvider;
  }
  
  class UploadRequest {
    constructor(
      private readonly storage: AttachmentStorage,
    ) {}
  
    async upload(attachment: Attachment): Promise<string> {
      return await this.storage.upload(attachment);
    }
  }
  
  async function handleUpload(
    company: Company,
    attachment: Attachment,
  ) {
    const storage =
      company.storage === "sftp"
        ? new SftpStorage()
        : company.storage === "webdav"
          ? new WebDavStorage()
          : new AwsStorage();
  
    return await new UploadRequest(storage).upload(attachment);
  }
  ```

  That single line, `new UploadRequest(storage)`, is dependency injection.

  The `UploadRequest` object is not reaching out to construct its own storage.
  Another part of the system chooses the dependency and hands it over. No
  decorator, container, or magic is required for the idea to be true.

  This is the mental model to keep:

  - One place chooses dependencies.
  - Another place uses dependencies.
  - Those two responsibilities get cleaner when they stop living in the same
    object.

  ## Husky DI gives the composition a home

  ```ts
  import {
    createContainer,
    createServiceIdentifier,
  } from "@husky-di/core";
  
  const IStorageProvider =
    createServiceIdentifier<StorageProvider>("IStorageProvider");
  const IAttachmentStorage =
    createServiceIdentifier<AttachmentStorage>("IAttachmentStorage");
  const IUploadRequest =
    createServiceIdentifier<UploadRequest>("IUploadRequest");
  
  const container = createContainer("HomepageTutorialContainer");
  
  container.register(IStorageProvider, {
    useValue: "sftp",
  });
  
  container.register(IAttachmentStorage, {
    useFactory: (currentContainer) => {
      const provider = currentContainer.resolve(IStorageProvider);
  
      if (provider === "sftp") {
        return new SftpStorage();
      }
  
      if (provider === "webdav") {
        return new WebDavStorage();
      }
  
      return new AwsStorage();
    },
  });
  
  container.register(IUploadRequest, {
    useFactory: (currentContainer) =>
      new UploadRequest(
        currentContainer.resolve(IAttachmentStorage),
      ),
  });
  
  const request = container.resolve(IUploadRequest);
  ```

  This is where `husky-di` enters the story. It does not invent DI. It gives
  the wiring a single, explicit home.

  On the homepage, that matters because it shows the real promise of the
  library:

  - identifiers give the runtime graph stable names
  - `register()` keeps the assembly rules visible
  - `resolve()` happens from a deterministic container boundary

  If you want even more automation later, `@husky-di/decorator` can sit on top.
  The important idea still stays the same: dependencies should be assembled at
  the edge, not hidden inside the workflow object.

  ## Testing gets easier because the puzzle pieces are swappable

  ```ts
  class FakeStorage implements AttachmentStorage {
    public uploaded: Attachment[] = [];
  
    async upload(attachment: Attachment): Promise<string> {
      this.uploaded.push(attachment);
      return attachment.id;
    }
  }
  
  async function testUploadRequest() {
    const storage = new FakeStorage();
    const request = new UploadRequest(storage);
  
    const attachment: Attachment = {
      id: "attachment-1",
      localPath: "/tmp/readme.txt",
      destination: "aws",
    };
  
    const uploadedId = await request.upload(attachment);
  
    console.log(uploadedId);
    console.log(storage.uploaded.length);
  }
  ```

  This is the payoff beginners feel the fastest.

  You do not need to patch private state or fake half the application just to
  test one path. You replace one dependency with a controlled implementation,
  keep the workflow intact, and verify behavior at the seam.

  That is why DI keeps showing up in good architecture conversations. It is not
  about ceremony. It is about making change, debugging, and testing feel
  mechanical instead of fragile.
