  ## Dependency injection starts before the container

  <DiWorkflowAnimation className="min-h-0 rounded-[1.5rem] bg-transparent px-0 py-0 shadow-none [&>div]:max-w-none [&>div]:rounded-[1.5rem] [&>div]:border-0 [&>div]:bg-transparent [&>div]:px-4 [&>div]:py-5 [&>div]:shadow-none [&>div]:backdrop-blur-none [&_pre]:text-[clamp(0.52rem,1.25vw,0.86rem)]" />

  Dependency injection usually makes more sense when you watch the flow before
  you stare at the API.

  A request comes in, something decides what implementation belongs in this
  environment, and only then does the workflow run with that dependency in
  place.

  That is the idea to hold onto for the rest of the page:

  - one place chooses dependencies
  - another place uses dependencies
  - the boundary between those two jobs is the seam DI gives you

  ## A feature can work and still be badly wired

  <HomepageTutorialCodeMarker stepId="hard-wired-branching" />

  The first real smell is not "missing framework." It is "one object owns too
  many decisions."

  In this version, the upload workflow is entangled with the logic for choosing
  AWS, SFTP, or WebDAV. The caller has to know which optional argument belongs
  to which branch, and the runtime decision happens deep inside the object that
  only wanted to upload a file.

  That is the core pressure that pushes people toward DI in the first place.

  ## First extract the behavior that changes

  <HomepageTutorialCodeMarker stepId="extract-capability" />

  Once the varying behavior gets a name, the rest of the design gets less
  mystical.

  `AttachmentService` does not actually care about vendors. It cares that
  *something* knows how to store an attachment. Pulling that capability into
  `AttachmentStorage` lets the workflow depend on an abstraction instead of
  three branches.

  Each implementation also becomes smaller and more honest: it keeps only the
  config and logic it truly owns.

  ## Injection just means passing the dependency in

  <HomepageTutorialCodeMarker stepId="inject-dependency" />

  This is the conceptual center of the whole tutorial.

  `UploadRequest` no longer decides what storage to construct. Another part of
  the system chooses the dependency first, then passes it into the workflow.

  No decorators or container are required for that idea to be true. DI is
  already happening as soon as dependency selection and dependency usage stop
  living in the same object.

  ## Husky DI gives the composition a home

  <HomepageTutorialCodeMarker stepId="container-composition" />

  This is where `husky-di` enters the story.

  The library does not invent dependency injection. It gives the wiring rules a
  single, explicit, inspectable home:

  - identifiers keep runtime names stable
  - `register()` makes assembly visible
  - `useClass` plus `resolve()` keeps the no-decorator core workflow compact

  So the code stays close to the mental model you just built, but the
  composition stops being scattered across request handlers and setup files.

  ## Testing gets easier because the puzzle pieces are swappable

  <HomepageTutorialCodeMarker stepId="testing-seam" />

  This is the payoff most people feel the fastest.

  Once the seam is explicit, you can replace a real dependency with a fake
  using the same container API. The test stays focused on the workflow
  contract, not on vendor behavior or hidden internal state.

  That is why dependency injection keeps coming up in good architecture
  conversations. It is not about ceremony. It is about making change,
  debugging, and testing feel mechanical instead of fragile.
